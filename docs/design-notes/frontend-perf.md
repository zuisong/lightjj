# Frontend Performance Notes

Rule-by-rule rationale behind the "Svelte Frontend Performance" checklist in [CLAUDE.md](../../CLAUDE.md#svelte-frontend-performance): the profiling evidence / war story behind each rule and the exemplar files that follow it, in the same order as the checklist. Most rules came out of profiling j/k keyboard navigation. A new lesson gets a one-line rule in the CLAUDE.md checklist AND a full entry here, never just one of the two.

How the three surfaces split: the CLAUDE.md checklist holds the always-loaded one-line rules; this file holds the per-rule rationale and is the rule-level source of truth; [ARCHITECTURE.md](../ARCHITECTURE.md#frontend-performance-patterns) repeats a few of these at the system level (prop scoping, caching layers, data-flow diagrams) and only needs updating when the architecture itself changes.

- **Don't use `:hover` for keyboard-navigable lists.** `:hover` recomputes on every paint — layout shift (error bar mount, `scrollIntoView`, post-mutation DOM reshuffle) slides it onto whatever row is now under a stationary mouse. Use JS-tracked `hoveredIndex` driven by `mousemove` (fires only on physical pointer movement, never on layout shift). See `RevisionGraph.svelte`: delegated `onmousemove` on container + `data-entry` attr + `class:hovered={hoveredIndex === line.entryIndex}`. Collapses sibling-chain `:has()` rules to one class selector when multiple rows belong to one logical item.
- **`untrack()` Svelte-4-store reads in effects that write back to the store.** Third-party libs shipping `Readable` stores with imperative setters (`setOptions`→internal `notify`→`writable.set`) self-loop when read in the same `$effect` that calls the setter. `const v = untrack(() => $store)` severs the dep; template reads stay reactive. Same pattern as `readByFile` in `diff-derivation.svelte.ts`. Better: don't use Svelte-4-store libs — `createWindower` (virtual.svelte.ts) uses runes directly and needs no untrack.
- **Debounce expensive work, not the selection state.** Update `selectedIndex` synchronously for instant visual feedback. Scheduling lives in `revision-navigator.svelte.ts`: `navigateCached` (cache hit → double-rAF) / `navigateDeferred` (miss → 50ms debounce). Double-rAF: rAF callbacks run BEFORE paint in the same frame (event → microtasks → rAF → style → layout → paint). Outer=frame-N-pre-paint, inner=frame-N+1-pre-paint — frame N paints cursor-only, diff DOM lands in N+1. Single rAF would batch cursor + diff into one frame, making cached j/k feel *slower* than uncached. Each navigate* cancels the prior schedule so rapid j/k only renders the destination. `navigateCached` bumps `revGen` at SCHEDULE time (not at rAF-fire time) so a `loadDiffAndFiles` suspended at `await api.revision()` can't resume in the gap and call `diff.load(stale)`. The `abort()` callback re-checks at fire time (cursor moved via a path that doesn't call navigate* — `loadLog`'s index reset, `selectRevisionCursorOnly`).
- **Opportunistic prefetch — only when current is cached.** `selectRevision` fires `prefetchRevision()` for the next revision in the navigation direction, but only when the current revision is already cached (instant main load → no network contention). With the batch `/api/revision` endpoint (1 req per prefetch, not 3) the 6-connection limit is no longer reachable, but the contention argument still holds: unconditional prefetch during rapid uncached j/k fires requests for revisions the user skips past. Fire-and-forget with swallowed errors.
- **Scope expensive `$derived` to minimal dependencies.** `workingCopyEntry = $derived(revisions.find(...))` only re-runs when `revisions` changes, not when `loading`/`mutating`/`diffLoading` flip. Nested inside `statusText` it was re-scanning 500 revisions 4-6× per mutation cycle.
- **Session-cache stable data.** `api.remotes()`/`api.aliases()` are promise-memoized — fetched once, reused for the session. Reset via `clearAllCaches()` (hard refresh). Error path clears the memo so retries work.
- **`createLoader()` factory** (`loader.svelte.ts`) encapsulates the generation-counter race-safe async pattern. Its `loading` flag is deferred via `setTimeout(0)` so cache hits (microtask-fast) never flip it — zero reactive updates on cached j/k navigation. `loader.set()` bumps generation so any in-flight `load()` is invalidated — "authoritative write wins". `loader.reset()` is `set(initial)`. **Prefer this over manual `diffLoading`/`diffError`/`diffGen` state** — the manual pattern has a subtle spinner-freeze bug when a no-fetch code path (early return) runs while a prior fetch is in flight (its `finally` gen-check fails to clear the flag). When migrating: side-effects-on-load that initialize *other* state (e.g. "set selectedX = result[0]") usually dissolve into `$derived(override ?? loader.value[0])` + nullable user-override — see GitModal selectedRemote. **Not a fit for barrier-gens** guarding per-key Map writes (e.g. DiffPanel.previewGen — bumped once per identity-change to invalidate ALL in-flight per-file fetches; no single `.value`).
- **Generation counters must cover WRITES, not just competing loads — symmetrically.** If `load()` has a gen-guard and `add()`/`update()`/`remove()` mutate the same `list` state, the mutators must (a) bump gen BEFORE await — otherwise `add()` completing mid-`load()` gets silently overwritten by load's final assignment; AND (b) check gen AFTER await — otherwise `load()` completing mid-`add()` means `add()` appends to the wrong revision's list. See `annotations.svelte.ts`: `const gen = bumpGen(); await ...; if (gen !== loadGen) return`. The asymmetry (bump without check) is the subtle trap.
- **Post-await identity guard is mandatory in inline-editor openers.** Every `await` is a j/k navigation window. `startDescriptionEdit`, `handleCommit`, `startEdit`, `saveFile` all capture `commit_id` before await, bail after if changed. The anti-pattern: opening a popup/editor over the NEW revision with OLD-revision prefill → mutation handler reads CURRENT `selectedRevision` → wrong commit gets wrong data. For bubble/modal state that survives across a save-time await (annotation bubble), capture `changeId`/`commitId` IN the state at open time — `diffTarget` is a fresh `$derived` object on every nav. **The guard cannot catch SYNCHRONOUS staleness** — `fetchPrefillDescription` reading `fullDescription` synchronously after `selectRevision` (which schedules via double-rAF) gets the PREVIOUS revision's loader value; the post-await guard then compares NEW===NEW (cid captured after select) and passes. Don't trust loader-derived state read in the same tick as `selectRevision` — fetch by commit_id instead (cached, so the hot path is still instant).
- **Keyboard/mouse parity: `diffFrozen` (from `ModeBase.diffFollows`) is the per-mode nav gate.** Previously mouse `onselect` and the multi-check `$effect` each spelled out `squash.active || split.active` to mean "diff frozen"; `inlineMode` was the trap (lumps rebase in). Now the mode factory declares `diffFollows` at creation (`modes.svelte.ts`), App derives `diffFrozen`, both sites read it. `handleInlineNav` still has a 3-way tuple for jk-dispatch (split has NO jk — `j`/`k` are hijacked by hunk-review cursor); that's a different question from "does diff follow". Grep the guard expression (`activeView = 'branches'`), not the fix site — `switchToBranchesView()` centralizes branches-entry semantics the way `switchToLogView()` does for log.
- **Effect declaration order + untracked reads = permanent skip.** Effects fire in declaration order within a batch. If effect-A (declared first) does `untrack(() => stateX)` for a dedup-check and effect-B (declared later) clears `stateX` as part of reset, then on a trigger that invalidates both: A reads the stale pre-reset `stateX`, dedup says "already have it", A skips work; B clears; A never re-fires (untracked). **Structural fix: use `createLoader` instead of hand-rolling** — the loader's generation counter doesn't care about effect order; stale resolves bounce regardless of what other effects did. `conflictFileDiffs` in DiffPanel was the exemplar (44-line hand-rolled effect, now `createLoader` + 11-line drive effect); any new per-revision async fetch should start there.
- **`{#key}` over manual reset effects.** When component state should reset on identity change, wrap the component in `{#key identityExpr}` in the parent — the component remounts with fresh state. Manual `$effect` + previous-value-tracking `let` is 7+ lines, fragile (sentinel collisions), and reinvents what Svelte gives you. Svelte 5.50+ flags the pattern as `state_referenced_locally`. See `EvologPanel` and `RevisionHeader` usage in `App.svelte`. **Key on what's DISPLAYED, not what's SELECTED** — `RevisionHeader`'s `{#key}` is `loadedTarget.changeId` (diff loader state), not `selectedRevision.commit` (cursor). Cursor can go null (custom revset excludes `@`) while loaded diff stays valid.
- **Guard `$derived` in hidden components.** `CommandPalette`'s `availableCommands` uses `if (!open) return []` to avoid recomputing when the palette is closed but its `commands` prop changes.
- **Split static and dynamic `$derived` arrays.** `commands` in App.svelte was rebuilding ~30 `PaletteCommand` objects on every check/uncheck (Space spam) because a few labels interpolate `checkedRevisions.size`. Now `staticCommands` (zero-dep `$derived.by` — thunk sidesteps TDZ for handlers declared below) + `dynamicCommands` (5 entries that actually need reactive labels) + `aliasCommands` → spread into final `commands`. Space-spam rebuilds 5 objects not 30.
- **`createDiffDerivation()` factory** (`diff-derivation.svelte.ts`) for per-file progressive computation. `run(files, cacheKey)` yields between files, publishes per-file, aborts on supersede, memoizes complete runs by cacheKey. `update(file)` for single-file deltas (context expansion) — preserves all other entries, aborts in-flight `run()`. `tryRestore(cacheKey)` for synchronous memo check — call it before deferring `run()` via setTimeout so memo hits restore zero-frame. `immediateBudget` option processes first N lines without yield to prevent plain-text flicker. Memo externalized via `readMemo`/`writeMemo` so multiple derivations can share one LRU bucket. **Internal `byFile` reads go through `readByFile()` (untracks)** — methods are called from `$effect` bodies; a naked `new Map(byFile)` would register the Source as a dep → `schedule_possible_effect_self_invalidation` → `effect_update_depth_exceeded` (Svelte 5.44+ batching change). Writes don't need untrack.
- **Defer `highlights.run()` via `setTimeout(0)` — but check `tryRestore` first.** The deferral lets the browser paint the selection highlight before any sync work runs. Memo hits skip the deferral: checking `tryRestore` synchronously in the `$effect` gives zero-frame restoration on revisit.
- **Cache derived computations by commit_id, not just the source data.** `derivedCache` (30-entry LRU keyed by `activeRevisionId`, in `diff-cache.ts` so it survives DiffPanel unmount) persists `highlightsByFile`/`wordDiffsByFile` so revisits restore instantly. Commit_id keying means rewrites auto-invalidate. Highlight HTML uses `tok-*` class names (not inline styles) → theme-agnostic → theme toggle is a pure CSS var swap, no cache invalidation.
- **`user-select: none`** on interactive lists prevents text selection artifacts during click/keyboard navigation.
- **Svelte 5 effects run after DOM updates** — no need for `requestAnimationFrame` to query updated DOM in `$effect`.
- **Hoist instance-invariant `$effect` to `<script module>`.** If an effect reads the same external state (`getComputedStyle`, `matchMedia`) across all instances and writes only to module-level state, per-instance is a mount-time perf trap — virtualized scroll through N rows = N× the work for identical results. Pattern: module `$state` + guard function (e.g. `refreshPalette(isDark)` no-ops if `paletteDark === isDark`); per-instance `$effect` just calls the guard. See `GraphSvg.svelte`.
- **Store stable keys in `$derived` Maps; compute equality at render.** `lineMatchMap` stored `isCurrent: index === currentMatchIdx` in values → entire Map rebuilt on Enter. Store `globalIndex` (stable), compute `globalIndex === currentMatchIdx` in the `{@const}` — only the 2 lines containing old/new-current re-render. Same principle as commit_id keying: the cache key should be the least-volatile thing that still identifies the data.
- **Fire-and-forget async in effects is fine** when the async function has its own error handling and generation counter for cancellation.
- **Skip word-diff for non-code files.** `shouldSkipWordDiff()` in DiffPanel skips LCS computation for SVG/XML/JSON/lock/map/minified suffixes and any file with >5000 diff lines (`WORD_DIFF_LINE_LIMIT`); per-line `MAX_TOKENS_FOR_LCS` guards pathological lines. Compute-skip is deliberately NOT tied to the 20k-char collapse limit — see the auto-collapse entry below.
- **Progressive word diffs.** The `wordDiffs` DiffDerivation yields between files and publishes per-file entries incrementally. Single-file context expansion calls `wordDiffs.update(file)` — only that file recomputes.
- **Auto-collapse large diffs — and keep collapse separate from compute-skip.** Collapse triggers: (1) any single file >500 lines (`AUTO_COLLAPSE_LINE_LIMIT`) starts collapsed; (2) any single file >20k chars (`AUTO_COLLAPSE_CHAR_LIMIT`) — catches minified JS on one long line; (3) total lines across all files >2000 (`AUTO_COLLAPSE_TOTAL_LINES`) collapses EVERYTHING — catches the "many moderate files" DOM-flooding case the per-file checks miss. Collapsed files render header-only via `{#if !isCollapsed}` in DiffFileView. Explicit collapse/expand intent is cached per change_id and overrides the live predicate per file. Expand-all is one toolbar click; Cmd+F search auto-expands matches. **Compute caps are separate** (v1.29): highlighting skips files >5000 lines (`HIGHLIGHT_SKIP_LINE_LIMIT`), word-diff skips >5000 lines, and the highlight run yields between files past `HIGHLIGHT_IMMEDIATE_LINES` (3000) — so a normal 600-line file that auto-collapses on chars still expands fully highlighted (the old conflation silently un-highlighted every ~300+-line code file). `HIDE_DIFF_TOTAL_LINES` (50k) / `HIDE_DIFF_FILE_LIMIT` (300 files) is the extreme fallback above which nothing renders until "Show anyway". Collapse is decided BEFORE render (`isFileCollapsed`, next entry) and explicit intent is change_id-cached; the old post-render auto-collapse $effect and its suppression flag are gone.
- **Collapse is decided before render; offscreen bodies defer their mount.** There is no post-render auto-collapse effect: `isFileCollapsed()` (DiffPanel) resolves `userExpanded > userCollapsed > auto predicate` at template time, so a new diff never transiently builds bodies it's about to collapse (the old effect paid full DOM construction on every navigation and only saved paint). Files past the ~600-line eager window render an estimated-height `.diff-body-placeholder` until a DiffPanel-owned IntersectionObserver (±2 viewports, one mount per frame, `holdViewport` re-anchoring for above-viewport mounts) or a programmatic jump mounts them. **Every programmatic jump to a file body must go through `revealFile(path)`** (expand-intent + mount) — querying body DOM after only toggling collapse state silently no-ops for deferred files. The observer effect must re-attach when the placeholder population can change (diff change, mount, collapse-intent flip) — missing the intent-set deps left expand-all's placeholders unobserved and nothing past the eager window ever mounted.
- **`content-visibility: auto` on `.diff-file`** — tells the browser to skip layout+paint for offscreen DiffFileView elements. The CSS fallback `contain-intrinsic-size: auto 200px` gives a height estimate for scroll calculations; DiffFileView additionally sets a per-file inline `contain-intrinsic-size: auto <est>px` computed from line count (suppressed for collapsed/preview/editing/binary states) so never-painted files reserve realistic scroll space. The `auto` keyword lets the browser remember the real height after first render so the scrollbar stays stable on revisit. Reduces the paint cost of large multi-file diffs.
- **Diff parser uses `b/` (destination) path** from `diff --git a/source b/destination` headers. The `a/` path is the source and can appear in multiple entries for copies/renames, causing duplicate `{#each}` keys.
- **Multi-check diffs go through `nav.loadMulti()`, not direct `diff.load()`.** The navigator advances `loadedTarget` + `diffContentKey` + `diff.value` as a TRIPLE — bypass it with `diff.load(multiTarget)` and the first two stay frozen at the prior single-revision key. DiffPanel's `contentMatchesTarget` invariant then compares fresh multi content against a stale single target, which renders highlights from the last single-rev view over the new multi diff. Any new code path that switches to/from multi must go through the navigator helper.
- **Tri-state `boolean | undefined` for UI gates derived from async-init state.** When a derived boolean depends on something loaded via `api.info()` / first fetch (e.g. `sshMode`, `hasFeatureX`), a `false` or `true` default enables or disables the wrong code path during the fetch window. Type as `boolean | undefined`, init `undefined`, have the `$derived` return the fail-safe explicitly for the undefined case. `editorConfigured` with `sshMode = false` default would have enabled open-in-editor during the brief pre-`loadInfo()` window for users who only configured `editorArgsRemote` (because `config.editorArgs.length === 0` on the wrong branch).
